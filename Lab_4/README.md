# Звіт: Дослідження поведінки allocator'а і сторожових сторінок

---
## Завдання 4.1 — Який максимальний розмір можна передати malloc(3) за один виклик?

Опис
- Параметр `malloc` має тип `size_t`. Теоретична максимальна величина — максимум `size_t` для платформи (на 64‑біті це `2^64−1`).
- Практична межа менша через реалізацію та внутрішні обчислення, які використовують знакові типи (консервативно `PTRDIFF_MAX = 2^63−1`).

Дії й перевірка
- Перевірили `sizeof(size_t)` та значення `PTRDIFF_MAX` (у програмі/через `sizeof` та `limits.h`).
- Запускали експерименти з викликами `malloc(very_large)` (без реального виділення пам’яті — лише перевірка поведінки).

Вивід
```bash
$ gcc -Wall -Wextra -O2 l4z1.c -o l4z1
$ ./l4z1
sizeof(size_t) = 8
size_t max theoretical = 18446744073709551615
PTRDIFF_MAX = 9223372036854775807
Conclusion: practical single-allocation limit ≈ 2^63-1 (≈8 EiB) on this libc/arch, not 2^64-1.
```

Висновок
- Теоретично `size_t` покриває до `2^64−1`, але реальна верхня межа одного виділення в поточній реалізації ≈ `2^63−1` байт (приблизно 8 ексбібайтів), оскільки malloc/менеджери пам’яті та обчислення зміщень використовують знакові типи або обмежують розміри.

## Завдання 4.2 — Що станеться, якщо передати malloc(3) від’ємний аргумент?

Опис
- Тестова програма обчислює `num = xa * xb`, де `xa`/`xb` — `int`. `num` використовується як аргумент для `malloc` після приведення до `size_t`.
- Перевіряли варіанти з переповненням signed int.

Запуск
```bash
gcc -Wall -Wextra -O2 l4z2.c -o l4z2
./l4z2 -100 17949672   
```

Вивід
```text
neg = -100, p1 = (nil)
num (signed overflow) = -1794967296
malloc(num as size_t) -> (nil)
```

Пояснення
- Якщо передати від’ємне signed значення, воно неявно перетвориться в дуже велике unsigned (`size_t`), але libc / kernel зазвичай відмовляють у такому виділенні і `malloc` повертає `NULL`.
- Якщо `num` обчислено в signed int і сталося переповнення, результат може бути від’ємним; при перетворенні це дасть непередбачувано великий `size_t` — але на практиці `malloc` повертає NULL або помилку (`ENOMEM`). Отже: не використовувати signed типи для розмірів, перевіряти переповнення заздалегідь.

Висновок
- Використовуйте `size_t` та перевірку переповнення перед множеннями розмірів (або використовуйте `reallocarray` / `calloc` для безпечного множення).

## Завдання 4.3 — Поведінка malloc(0)

Опис
- Написали простий тест: виклик `malloc(0)`, перевірка чи `NULL` і чи можна передати отримане значення у `free()`, запуск під `ltrace`.

Результати
```bash
gcc -Wall -Wextra -O2 l4z3.c -o l4z3
ltrace ./l4z3
```

Приклад виводу
```text
malloc(0) returned non-NULL pointer: 0x556b9f3d92a0
free(0x556b9f3d92a0) = <void>
+++ exited (status 0) +++
```

Пояснення
- Стандарт дозволяє реалізації повертати або `NULL`, або ненульовий вказівник для `malloc(0)`. Обидва варіанти коректні; якщо отримано ненульовий вказівник, його дозволено звільнити через `free()`. Не покладатися на конкретну поведінку — обробляти `size==0` спеціально, якщо потрібна однозначність.

Висновок
- `malloc(0)` може повернути `NULL` або валідний вказівник; перевіряйте семантику в коді і обробляйте випадок `size==0` окремо.

## Завдання 4.4 — Помилки у фрагменті з malloc/free; демонстрація use-after-free

Фрагмент коду (помилковий)
```c
void *ptr = NULL;
while (<cond>) {
    if (!ptr)
        ptr = malloc(n);
    ... use ptr ...
    free(ptr);
}
```

Проблема
- Після `free(ptr)` вказівник не обнуляється. На наступній ітерації `ptr` не `NULL`, перевірка `if (!ptr)` не виконається, але `ptr` вказує на звільнену пам’ять → use-after-free / подвійний free.

Тестові програми
- `l4z4_bad.c` — демонструє помилку (не обнуляє `ptr`).
- `l4z4_good.c` — після `free(ptr)` робить `ptr = NULL` (або змінює логіку), що усуває проблему.

Приклад запуску / спостереження
```bash
gcc l4z4_bad.c -o l4z4_bad && ./l4z4_bad    # може падати або показати некоректні дані
gcc l4z4_good.c -o l4z4_good && ./l4z4_good  # працює коректно в циклі
```

Висновок
- Після `free(ptr)` завжди або обнуляйте вказівник (`ptr = NULL`), або організуйте життєвий цикл пам’яті так, щоб не використовувати вже звільнений блок.

## Завдання 4.5 — Що відбувається, якщо realloc(3) не може виділити пам’ять?

Опис
- Тест: виділення невеликого блока, потім спроба `realloc` до дуже великого розміру; присвоєння в тимчасову змінну `tmp = realloc(ptr, huge)` і перевірка.

Вивід
```text
Before realloc: ptr=0x55a3..., data=ok
realloc failed: tmp = (nil), errno = 12 (ENOMEM)
Old ptr still valid: ptr=0x55a3..., data=ok
```

Пояснення
- `realloc` при невдачі повертає `NULL`, але старий `ptr` лишається дійсним (не звільняється). Якщо робити `ptr = realloc(ptr, huge)` без перевірки, ви втратите доступ до старого блоку (витік) або перепишете `ptr = NULL` і втратите дані. Тому завжди використовувати тимчасову змінну:

```c
void *tmp = realloc(ptr, newsize);
if (!tmp) { /* обробити помилку, ptr лишається валідним */ }
else { ptr = tmp; }
```

Висновок
- Перевіряйте результат `realloc` через тимчасову змінну; не присвоюйте безпосередньо в оригінальний вказівник.

## Завдання 4.6 — realloc(NULL, size) і realloc(ptr, 0)

Опис та результати
- `realloc(NULL, size)` еквівалентний `malloc(size)` — виділяється новий блок.
- `realloc(ptr, 0)` залежить від реалізації: у нашому середовищі `realloc(ptr, 0)` повернув `NULL` і звільнив старий блок (поведінка libc: може повернути `NULL` або ненульовий вказівник — не покладатися на одну поведінку).

Приклад запуску
```bash
gcc l4z6.c -o l4z6 && ./l4z6
realloc(NULL, 5) -> 0x55a...
realloc(b, 0) -> (nil)   # old block freed
```

Рекомендація
- Якщо потрібно явно звільнити пам’ять, використовуйте `free(ptr)` і `ptr = NULL`; не покладайтеся на `realloc(ptr, 0)` для переносів або специфічних семантик.

## Завдання 4.7 — Переписати код, використовуючи reallocarray(3)

Оригінал
```c
struct sbar *ptr, *newptr;
ptr = calloc(1000, sizeof(struct sbar));
newptr = realloc(ptr, 500*sizeof(struct sbar));
```

Переробка з `reallocarray`
```c
struct sbar *ptr, *newptr;
ptr = calloc(1000, sizeof(struct sbar));
newptr = reallocarray(ptr, 500, sizeof(struct sbar));
```

Пояснення
- `reallocarray(count, size)` захищає від переповнення при множенні `count*size` і є безпечнішою альтернативою прямому множенню.
- Тестовий запуск показав очікувані байти:
  - for 1000 elements: 16000 bytes (наприклад)
  - after `reallocarray` to 500: 8000 bytes

`ltrace` підтвердив виклики `calloc`, `reallocarray`, `free`.

Висновок
- Для безпечного масштабування масивів використовуйте `reallocarray`/`callocarray`, щоб уникнути silent overflow при множенні розмірів.

## Завдання 4.8 — Використання mmap() та mprotect() для сторожових сторінок. (5 варіант)

Опис
- Виділяємо 3 сторінки: `[guard][usable][guard]`. Середню сторінку робимо `PROT_READ|PROT_WRITE`, крайні — `PROT_NONE`.
- Пишемо всередині буфера (успішно). Пишемо за кінець буфера — звернення потрапляє у guard page → CPU exception → `SIGSEGV` (гарантований секфолт).

Мета
- Показати, як створити сторожові сторінки (guard pages) для гарантування `SIGSEGV` при доступі поза межами буфера.

Підготовка та компіляція
```bash
gcc -Wall -Wextra -O2 l4z8.c -o l4z8
```

Запуск
```bash
./l4z8
```

Очікувана поведінка
- Програма виводить адреси відображення (base, mid, guards).
- Виконує запис у межах середньої сторінки — успіх.
- Виконує запис на 1 байт за кінцем середньої сторінки — спричиняє `SIGSEGV` (Segmentation fault).

Налагодження
- `gdb`:
  - `gdb ./l4z8`
  - `run`
  - після `SIGSEGV`: `bt`
- `strace`:
  - `strace -f ./l4z8`  (поки що покаже `mmap`/`mprotect` і signal)

```
Запуск

```bash
$ gcc -Wall -Wextra -O2 l4z8.c -o l4z8
$ ./l4z8
pagesize = 4096
mapping base = 0x7f9a2c800000
left guard page = 0x7f9a2c800000
usable buffer (mid) = 0x7f9a2c801000
right guard page = 0x7f9a2c802000
writing inside buffer at offset 4095 ...
ok: wrote inside buffer
writing just past buffer (this should cause SIGSEGV)...
Segmentation fault (core dumped)
```

Пояснення
- `mprotect(..., PROT_NONE)` робить сторінку недоступною; будь-яка операція доступу спричиняє `SIGSEGV` на рівні CPU/ядра → гарантований segmentation fault.
