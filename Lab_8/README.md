## Завдання 1 — Частковий запис у write()
Проблема
- Системний виклик write(fd, buffer, nbytes) може повернути значення, менше ніж nbytes.
- У прикладі: nbytes = 200000, реально записано count = 65536.

Причина
- Дескриптор відкрито в неблокуючому режимі (O_NONBLOCK) або канал (pipe) має обмежений буфер, тому системний буфер не вмістив усі дані за один виклик.

Рішення
- Після кожного write() перевіряємо повернуте значення.
- Якщо записано менше байтів — у циклі дозаписуємо решту (зсув буфера + зменшення залишку).
- Обробляємо помилки: EINTR, EAGAIN / EWOULDBLOCK (на неблокуючому fd — повторна спроба або очікування).
- У прикладі додатково показана демонстрація write() у pipe у неблокуючому режимі, що дає частковий запис.

Ключові моменти:
- Завжди використовувати цикл для допису: while (remaining > 0) { w = write(...); handle errors; advance pointer; remaining -= w; }

## Завдання 2 — Читання з довільної позиції
Вхідний файл містить байти:
`4, 5, 2, 2, 3, 3, 7, 9, 1, 5`

Операції
- Виклик `lseek(fd, 3, SEEK_SET)` встановлює позицію на зміщення 3 (четвертий байт).
- Виклик `read(fd, buffer, 4)` читає 4 байти від цієї позиції.

Результат
- Зчитані байти: `2 3 3 7` — саме такі значення опинились у буфері.

Реалізація: l8z2.c — створює тестовий файл з вказаними байтами, виконує lseek та read, виводить прочитані значення.

## Завдання 3 — Дослідження швидкості qsort()
Мета
- Дослідити поведінку стандартного qsort() для різних структур вхідних масивів і виміряти середню кількість порівнянь й середній час сортування.

Типи вхідних масивів
- випадковий;
- вже відсортований (зростання);
- зворотний (спадання);
- усі елементи однакові.

Тести коректності
- Проведені тести сортування для кожного типу — усі пройшли успішно:
  - змішаний масив: УСПІХ
  - відсортований: УСПІХ
  - зворотний: УСПІХ
  - однакові: УСПІХ

Результати експерименту
- випадковий: сер. кількість порівнянь = 718 231, сер. час = 5.141 мс  
- вже відсортований: сер. кількість порівнянь = 382 512, сер. час = 1.288 мс  
- зворотний: сер. кількість порівнянь = 401 952, сер. час = 1.663 мс  
- усі однакові: сер. кількість порівнянь = 382 512, сер. час = 1.333 мс

Висновок
- Найповільнішим для цього виміру виявився випадковий масив. Продуктивність qsort() значною мірою залежить від структури вхідних даних.

Реалізація: l8z3.c — підрахунок порівнянь у comparator'і, вимір часу (clock_gettime), середні значення по кількох прогонів.

## Завдання 4 — Робота fork()
Поведінка
- Виклик fork() створює два процеси: батьківський і дочірній.
- Значення, яке повертає fork():
  - у дочірньому процесі: `0`
  - у батьківському процесі: PID дочірнього процесу (додатне число)

Вивід:
```
67163
0
```
Реалізація: l8z4.c — викликає fork(), робить printf("%d\n", pid) для значення повернення fork(), щоб демонструвати різницю.

## Завдання 5 — Симуляція збою під час write() та збереження цілісності даних (варіант 5)
Мета
- Імітувати ситуацію часткового запису (збій у середині write()) і продемонструвати механізм, який гарантує цілісність даних.

Опис підходу (реалізація в l8z5.c)
1. Симуляція часткового запису
   - Використано неблокуючий канал (pipe) з обмеженим буфером або велику передачу у неблокуючий дескриптор, так що write() повертає менше, ніж запитано (наприклад, 65536 з 200000 або 1 МБ).
   - Програма намагається записати великий блок даних (наприклад, 1 МБ) у канал; write() може повернути частковий запис.
   - Реалізовано цикл дозапису: програма зберігає зміщення і продовжує запис, доки всі байти не передані.

2. Перевірка цілісності
   - Перед записом обчислюється контрольна сума CRC32 для всіх даних.
   - Після повного запису програма читає назад дані з каналу/файлу (залежно від реалізації) і обчислює CRC32 знову.
   - Порівняння контрольних сум підтверджує, що жоден байт не загублено і дані не пошкоджено.

Ключові моменти реалізації
- Встановлення O_NONBLOCK (fcntl) для демонстрації часткового write().
- Надійний цикл запису: advance pointer, зменшення залишку, повторні виклики write().
- Обробка помилок: EINTR, EAGAIN/EWOULDBLOCK.
- Використання CRC32 для валідації цілісності даних після передачі.

Результат
- Програма демонструє, як правильно обробляти часткові записи і забезпечити цілісність даних навіть при «збої» посеред операції.
