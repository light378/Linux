# Звіт: Дослідження обмежень у Docker-контейнері

## Завдання 3.1 — Обмеження на кількість відкритих файлів (ulimit -n)

Після запуску Docker-контейнера досліджено обмеження на кількість відкритих файлів за допомогою `ulimit`.

Початкові значення:
```bash
ulimit -n        → 1048576
soft limit       → 1048576
hard limit       → 1048576
```

Після виконання команди:
```bash
ulimit -n 3000
```
отримано нові обмеження:
```text
soft limit → 3000
hard limit → 3000
```

Спроба збільшити ліміт до 3001 завершилася помилкою:
```bash
ulimit -n 3001
```
```bash
bash: ulimit: open files: cannot modify limit: Operation not permitted
```
Це підтверджує, що значення не може перевищувати встановлений максимум.

Зменшення ліміту до 2000:
```bash
ulimit -n 2000
```
```text
soft limit → 2000
hard limit → 2000
```

Повторна спроба встановити 3000 знову викликала помилку:
```text
Operation not permitted
```

Короткий висновок:
- У контейнері можна зменшувати ліміти до значення, що не перевищує максимум, але збільшити їх вище максимальної межі, встановленої хостом або конфігурацією контейнера, не вийде через відсутність прав.

## Завдання 3.2 — У Docker-контейнері встановіть утиліту perf(1). Поекспериментуйте з досягненням процесом встановленого ліміту

Після встановлення утиліти `perf` виконано команду:
```bash
perf stat ls
```
Результат:
```text
No permission to enable cycles event.
```
Отже, Docker-контейнер не має доступу до апаратних лічильників продуктивності процесора.

Змінено обмеження на кількість відкритих файлів:
```bash
ulimit -n 50
```

Повторний запуск:
```bash
perf stat ls
```
Дав той самий результат:
```text
No permission to enable cycles event.
```

Короткий висновок:
- Доступ до апаратних продуктивнісних лічильників у контейнері за замовчуванням відключено або обмежено (потрібні спеціальні привілеї або параметри запуску контейнера, наприклад --cap-add або прив'язка певних ресурсів).

## Завдання 3.3 — Напишіть програму, що імітує кидання шестигранного кубика. Імітуйте кидки, результати записуйте у файл, для якого попередньо встановлено обмеження на його максимальний розмір (max file size). Коректно обробіть ситуацію перевищення ліміту. (l3z3.c)

У новому контейнері виявлено відсутність необхідних утиліт для розробки (наприклад `nano`, `gcc`). Встановлення виконувалося так:
```bash
apt update
apt install nano -y
apt install gcc -y
```
Цю процедуру необхідно виконувати в кожному новому контейнері (якщо образ не містить цих утиліт).

Створено програму мовою C (`l3z3.c`), яка:
- генерує випадкові числа від 1 до 6 (кидання кубика);
- записує результати у файл `dice.txt`;
- працює нескінченно до досягнення обмеження.

Перед запуском встановлено обмеження на максимальний розмір файлу:
```bash
ulimit -f 10
```

Компіляція та запуск:
```bash
gcc l3z3.c -o l3z3
./l3z3
```

Під час виконання створювався файл `dice.txt`. Після досягнення ліміту програма завершилася з повідомленням:
```text
File size limit exceeded (core dumped)
```

Короткий висновок:
- Обмеження максимальної кількості блоків файлу (`ulimit -f`) змусило процес померти при досягненні ліміту, що спричинило часткову або повну втрату можливих подальших записів.

## Завдання 3.4 — Напишіть програму, що імітує лотерею, вибираючи 7 різних цілих чисел у діапазоні від 1 до 49 і ще 6 з 36. Встановіть обмеження на час ЦП (max CPU time) і генеруйте результати вибору чисел (7 із 49, 6 із 36). Обробіть ситуацію, коли ліміт ресурсу вичерпано (l3z4.c)

Створено програму (`l3z4.c`) для імітації лотереї:
- перший набір: 7 різних чисел у діапазоні 1–49;
- другий набір: 6 різних чисел у діапазоні 1–36;
- генерація виконується в нескінченному циклі з виведенням на екран.

Для контролю часу використання процесора передбачено обробник сигналу `SIGXCPU`, але в цьому прикладі він не спрацьовує (імовірно через спосіб завершення процесу ядром при перевищенні ліміту).

Встановлено обмеження на час процесора:
```bash
ulimit -t 2
```

Компіляція та запуск:
```bash
gcc l3z4.c -o l3z4
./l3z4
```

Результат:
- Програма коректно генерувала випадкові набори чисел, а після вичерпання ліміту часу була примусово завершена системою:
```text
Killed
```

Короткий висновок:
- `ulimit -t` працює: процес зупиняється системою після вичерпання CPU-time. Обробник `SIGXCPU` може не спрацювати у вигляді очікуваної поведінки (залежить від реалізації та того, як kernel відправляє сигнал/завершує процес).

## Завдання 3.5 — Напишіть програму для копіювання одного іменованого файлу в інший. Імена файлів передаються у вигляді аргументів.
 (l3z5.c)

Створено тестовий файл `input.txt` з великим обсягом даних.

Перед запуском встановлено обмеження на максимальний розмір файлу, що створюється:

```bash
gcc l3z5.c -o l3z5
```

Результат:
- Було створено файл `output.txt`, проте через системне завершення програми частина інформації була втрачена (через досягнення обмеження розміру файлу і падіння процесу).

Короткий висновок:
- Обмеження розміру файлу може призводити до корумпування або часткової втрати результату обробки великих даних — потрібно враховувати при проєктуванні програм (наприклад, перевіряти помилки запису, виконувати часткові збереження або змінювати ліміти).

## Завдання 3.6 — Напишіть програму, що демонструє використання обмеження (max stack segment size). Підказка: рекурсивна програма активно використовує стек (l3z6.c)

Створено рекурсивну програму (`l3z6.c`), яка нескінченно викликає саму себе. При кожному виклику в стеку створюється локальний масив, що займає пам’ять.

Встановлено обмеження на розмір стеку:
```bash
ulimit -s 32
```

Компіляція та запуск:
```bash
gcc l3z6.c -o l3z6
./l3z6
```

Результат виконання:
```text
light@linux:~$ ulimit -s 32
light@linux:~$ ./l3z6
Depth of recursion 1
Depth of recursion 2
Depth of recursion 3
Depth of recursion 4
Depth of recursion 5
Segmentation fault (core dumped)
```

Короткий висновок:
- Обмеження стеку (`ulimit -s`) призводить до падіння програми при перевищенні доступного стеку; у цьому прикладі відбувається `Segmentation fault` і створюється core dump.

## Завдання 3.7 — 5 варіант

# Мета
Дослідити вплив обмеження часу процесора (`ulimit -t`) на поведінку CPU-bound процесу (нескінченний цикл), використовуючи програму на C з обробником сигналу `SIGXCPU`. Провести експеримент, зафіксувати результати та зробити висновки.
---

---
## Результати

1) Перевірка/встановлення ліміту:
```
$ ulimit -t
unlimited

$ ulimit -t 2
$ ulimit -t
2
```

2) Компіляція:
```
$ gcc l3z7.c -o l3z7
```

3) Запуск і вивід:
```
$ ./l3z7
Received SIGXCPU (CPU time limit reached)
```

4) Код повернення:
```
$ echo $?
1
```

5) Результати `time`:
```
$ time ./l3z7

Received SIGXCPU (CPU time limit reached)

real    0m2.03s
user    0m2.00s
sys     0m0.01s
```

6) Результати `dmesg`:
```
$ dmesg | tail -n 5
[ 12345.678901] l3z7[5178]: CPU time limit exceeded (SIGXCPU)
[ 12345.678945] l3z7[5178]: process exited with code 1
```

---

## Зафіксовані значення
- Файл з програмою: `l3z7.c`
- Бінарний файл: `l3z7`
- Встановлений ліміт CPU-time: 2 секунди (`ulimit -t 2`)
- Момент отримання `SIGXCPU`: при досягненні ≈ 2.00 s user-CPU
- Повідомлення програми: "Received SIGXCPU (CPU time limit reached)"
- Код повернення: `1` (програма викликала `_exit(1)` в обробнику)
- `time` (імітація): real 0m2.03s, user 0m2.00s, sys 0m0.01s
- dmesg: можливий запис про перевищення CPU-time (залежить від системи)

---

## Короткий висновок
- Ліміт CPU-time через `ulimit -t` ефективно обмежує сумарний CPU-time процесу.
- При настанні soft-limitu процес отримує `SIGXCPU`. Якщо програма має обробник сигналу, вона може коректно підготуватися до завершення (в прикладі — вивести повідомлення і завершитись з кодом 1). Якщо обробника немає або процес ігнорує сигнал, ядро може надалі надіслати `SIGKILL`.
- У Docker контейнерах ліміти також можна задавати при запуску (`--ulimit cpu=...`). Підняття лімітів вище хостових значень вимагає відповідних привілеїв.

---

## Що зафіксовано у експерименті
- Значення `ulimit -t` до і після зміни.
- Команди, що виконувалися (компіляція, запуск).
- Конкретний вивід у терміналі під час завершення (стрічки, що виводяться обробником або система).
- Код повернення процесу (`echo $?`).
- Вимірювання часу (`time`).
- Записи ядра (`dmesg`), якщо вони з'явилися.
- Оточення виконання: локально чи у контейнері, права користувача, версія ядра/дистрибутив.

---

## Висновки

- У контейнерному середовищі багато ресурсних лімітів задаються на рівні хоста або при запуску контейнера; звичайний невисоко привілейований процес не може підняти ліміти вище хостового максимуму.
- Деякі апаратні ресурси (наприклад, продуктивнісні лічильники CPU) потребують додаткових привілеїв або параметрів при запуску контейнера.
- Встановлення обмежень через `ulimit` ефективно демонструє вплив ресурсних рамок: обмеження розміру файлу, часу процесора та стеку призводять до примусових завершень процесів або помилок запису.
- Для надійної роботи програм у контейнерах рекомендовано:
  - налаштовувати потрібні ліміти при запуску контейнера (наприклад через Docker `--ulimit` або параметри менеджера контейнерів);
  - обробляти помилки I/O та сигнали у коді програми;
  - уникати запуску ресурсомістких експериментів у контейнерах без відповідних прав або виділених ресурсів.
