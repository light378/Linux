# Лабораторна робота №9. Керування правами доступу в Linux

## Завдання 1. Аналіз облікових записів системи

Програма отримує перелік користувачів через `getent passwd`, зчитує поріг `UID_MIN` з файлу `/etc/login.defs` і перевіряє, чи існують облікові записи з `UID >= UID_MIN`, окрім поточного користувача.

**Результат виконання:**  
- `UID_MIN = 1000`  
- Знайдено запис: `nobody (UID=65534)`

Отже, у системі є принаймні один «системний» користувач, який не є поточним, але за формальним порогом вважається «звичайним».

---

## Завдання 2. Виконання команди з підвищеними правами

Програма запускає `cat /etc/shadow` за допомогою `sudo` (сама програма виконується від звичайного користувача).

**Результат:**  
- Запит пароля для `sudo`  
- Після введення коректного пароля виводиться вміст `/etc/shadow`

Це демонструє можливість підвищення привілеїв у межах дозволеної політики `sudo`.

---

## Завдання 3. Взаємодія з файлом, створеним від root

Програма створює файл від звичайного користувача, потім копіює його в домашній каталог через `sudo` (файл стає власністю `root`). Далі перевіряються дії від імені звичайного користувача:

- **Спроба змінити файл** → `Permission denied` (немає прав запису у файл)
- **Видалення через `rm`** → Запит підтвердження (файл захищений від запису), після підтвердження видалення успішне.

**Пояснення:**  
Право на видалення файлу в каталозі визначається правами **самого каталогу** (потрібен `w` і `x` для каталогу). Файл може бути незмінним, але його можна видалити, якщо каталог дозволяє змінювати вміст.

---

## Завдання 4. Визначення поточного користувача та його груп

Програма послідовно виконує команди `whoami` та `id`.

**Результат:**
```
whoami: light
id: uid=1000(light) gid=1000(light) groups=1000(light),27(sudo),125(docker)
```

**Висновок:**  
Команда `id` показує не лише UID/GID, а й усі групи, до яких належить користувач (включаючи додаткові, наприклад `sudo` та `docker`).

---

## Завдання 5. Зміна власника та прав доступу

Програма створює тимчасовий файл від звичайного користувача, а потім за допомогою `sudo` змінює його власника та права. Після кожної операції перевіряється можливість читання й запису від звичайного користувача.


                
`chown root:root` + `chmod 600`        
`chmod 666` (власник root)              
Повернення власника користувачу + `chmod 600`

**Висновок:**  
Доступ до файлу визначається комбінацією власника та бітів прав. Навіть root-файл може бути доступним для читання/запису звичайним користувачем, якщо права дозволяють (наприклад, `666`).

---

## Завдання 6. Дослідження прав у різних каталогах

Програма виконує `ls -l` для трьох каталогів:  
- `~/` (домашній)  
- `/usr/bin`  
- `/etc`  

Потім перевіряє можливість читання, запису та виконання для обраних файлів.

**Результати:**

```
home/light/lab9_6_demo.txt
/usr/bin/ls           
/etc/passwd          
/etc/shadow
```
          
**Висновок:**  
Права доступу чітко розмежовують дії. Без підвищення привілеїв не можна отримати доступ до захищених файлів (наприклад, `/etc/shadow`), навіть для читання.

---

## Завдання 7. Лише права на запис, читання заборонено

### Як це можливо?
Файл має права, наприклад, `0200` (`--w-------`). Власник (або будь-який інший користувач, якщо встановлено `222`) може **записувати** дані, але **читати** – ні. Це досягається зняттям біта `r` і залишенням `w`.

### Наслідки
- Програма може писати логи, але не може їх перевіряти – ускладнює налагодження.
- Можливе псування даних (перезапис) без можливості побачити попередній вміст.
- Деякі системні механізми (наприклад, `sudo` та деякі FIFO) використовують такий режим для захисту конфіденційності.
